# AGENTS.md

## 文档维护规则

### 文档索引

- `docs/index.md` 是文档入口。新增、删除、重命名 `docs/*.md` 时必须同步更新索引。
- 索引按主题分组，不按文件名字母序堆列表。
- 每个索引项用一句话说明文档解决什么问题，避免只放链接。

### Doc Lint

修改 Markdown 文档后，至少做以下检查：

- 标题检查：每篇文档只能有一个一级标题，并且一级标题必须在第一行。
- 链接检查：相对链接必须能指向真实文件；新增专题文档必须从 `docs/index.md` 或相关上游文档可达。
- 结构检查：长文档要有清晰的二级标题；不要把多个主题塞进一个无分节段落。
- 术语检查：首次出现的英文术语应给出中文解释；复杂术语需要一句解释或例子。
- 范围检查：文档标题是什么，正文就聚焦什么；不要把项目实现决策混进纯概念介绍文档。
- 引用检查：涉及外部标准、格式、协议、库能力时，优先引用官方资料，并在文末保留参考资料链接。
- 信息密度检查：不要写“片汤话”。每篇概念文档开头必须用 1-3 段说明本质：它解决什么问题、输入/输出是什么、和相邻概念有什么边界。
- 机制检查：介绍协议、格式、库、软件或标准时，必须说明它在链路中的位置，以及核心机制；不能只写“用于 X、适合 Y、生态很大”。
- 对比检查：容易混淆的概念必须显式对比，例如 MIDI vs 音频、synth vs sampler、plugin format vs MIDI、timecode vs clock。
- 例子检查：复杂概念必须给一个具体例子，优先使用最小可运行链路、消息样例、文件片段或实际场景。
- 取舍检查：写优缺点时必须说明原因和适用条件，不要只列“优点/缺点”名词。
- mdv 检查：处理 mdv 批注后必须用 `mdv comments reply` 或 `mdv comments reply-batch` 回复；不要只改文档不回复批注。

### 文档写作标准

写文档时优先回答这几个问题：

1. 它到底是什么，不是什么。
2. 它解决什么具体问题。
3. 它在完整链路里的上游和下游是谁。
4. 它的核心数据、消息、接口或机制是什么。
5. 它和相邻概念的区别是什么。
6. 一个最小例子长什么样。
7. 它的限制在哪里，什么时候不该用。

禁止只写空泛介绍，例如：

```text
X 是一种常见工具，适合 Y，生态很大，使用广泛。
```

应该改成：

```text
X 位于 A 和 B 之间，负责把 C 转成 D。它接收的数据是 ...，输出的是 ...。
和 Z 的区别是 ...。例如 ...
```

推荐命令：

```bash
find docs -maxdepth 1 -type f -name '*.md' -print | sort
rg -n '^# ' docs/*.md
rg -n '\\[[^]]+\\]\\(([^)#]+\\.md)(#[^)]+)?\\)' docs/*.md
```
