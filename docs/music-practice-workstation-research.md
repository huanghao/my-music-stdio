# 音乐练习工作站设计调研草案

## 目标

这个项目不是单一的谱面编辑器、伴奏生成器或声卡练习工具，而是一个面向多乐器练习的音乐工作站。

核心目标：

- 显示和编辑多种谱面：六线谱、五线谱、简谱、口琴吹奏谱。
- 支持谱面之间的转换。
- 支持不同调、调式、移调、变调夹、口琴 position 等分析。
- 支持和弦编辑与和声分析。
- 支持按风格自动生成伴奏。
- 支持声卡输入，例如吉他接入后进行调音、跟练、音准和节奏检测。

## 总体架构思路

建议采用统一的内部音乐模型作为系统核心。

```text
Music Model
  -> 五线谱
  -> 六线谱
  -> 简谱
  -> 口琴谱
  -> 和弦图
  -> MIDI/音频播放
  -> 伴奏生成
  -> 声卡练习分析
```

谱面格式不应该各自成为源数据。五线谱、六线谱、简谱、口琴谱都应该是同一个音乐模型的不同视图。

这样设计的原因：

- 避免不同谱面之间互转时出现大量特殊逻辑。
- 方便后续支持导入导出 MusicXML、MIDI、Guitar Pro、ChordPro 等格式。
- 方便把和声分析、移调、伴奏生成、练习评分统一接入。
- 可以逐步增加乐器指法模型，而不破坏基础数据结构。

## 内部音乐模型需要表达的内容

### 基础音乐信息

- 音高：绝对音高、音名、唱名、调内级数。
- 节奏：拍号、时值、附点、连音、切分、弱起、小节线。
- 速度：BPM、速度变化、ritardando、accelerando。
- 力度：强弱记号、渐强、渐弱。
- 装饰音：倚音、颤音、滑音、琶音等。

### 和声信息

- 和弦名称，例如 `Cmaj7`、`Dm7`、`G7`。
- 和弦组成音。
- 转位。
- 和弦功能，例如主、下属、属。
- 罗马数字分析。
- Nashville Number System。
- 调性、调式、临时转调。
- 借用和弦、二级属和弦等扩展分析。

和弦表示法本身需要单独建模。常见输入包括流行/爵士和弦符号、罗马数字分析、Nashville Number System、级数和弦、固定调音名等；它们解决的是不同问题，不能简单合并成一个字符串。建议先学习并支持流行/爵士和弦符号的基础语法，再把罗马数字和 Nashville Number System 作为分析层或移调层表示。详见 [和弦表示法调研](chord-symbol-notation-research.md)。

### 乐器演奏信息

吉他：

- 弦、品、调弦。
- 变调夹。
- 指型。
- 开放弦。
- 推弦、滑音、击弦、勾弦、泛音、闷音。

口琴：

- 口琴类型：十孔布鲁斯口琴、复音口琴、半音阶口琴。
- 调性。
- 孔位。
- 吹、吸。
- 压音、超吹、超吸。
- position，例如 1st position、2nd position、3rd position。

小提琴：

- 弦。
- 把位。
- 指法。
- 换弦。
- 空弦。
- 连弓、分弓。

钢琴：

- 左右手。
- 声部。
- 和弦分解。
- 伴奏织体。
- 踏板信息。

### 曲式结构

- Intro。
- Verse。
- Chorus。
- Bridge。
- Solo。
- Ending。
- Repeat。
- Segno、Coda。
- 小节范围和段落标签。

## 谱面和文件格式调研

### MusicXML

MusicXML 是传统乐谱交换格式，适合导入导出五线谱、多声部、多谱表和常见谱面标记。它应作为外部交换格式接入，导入后转换成自己的 Music Model，而不是直接当内部模型。详见 [MusicXML 格式调研](musicxml-format-research.md)。

### MIDI

MIDI 适合播放、录制、伴奏生成和外部设备输入。它描述演奏事件和控制信息，不适合作为谱面源格式，因为缺少清晰的谱面语义，例如连音、声部、指法、和弦标注。详见 [MIDI 格式调研](midi-format-research.md)。

### Guitar Pro 格式

Guitar Pro 格式是吉他谱生态的核心入口，覆盖六线谱、技巧记号和播放信息。第一阶段建议依赖现成解析能力，只验证导入覆盖率和技巧映射，不完整复刻文件格式。详见 [Guitar Pro 格式调研](guitar-pro-format-research.md)。

### ABC Notation

ABC Notation 是文本化旋律记谱，适合快速录入简单旋律、调号、拍号、节奏和和弦标记。它对复杂谱面、吉他技巧和口琴技巧支持有限，应作为轻量输入格式而非完整谱面格式。详见 [ABC Notation 调研](abc-notation-research.md)。

### ChordPro

用途：

- 歌词 + 和弦。
- 民谣弹唱、和弦谱、歌词排版。

需要重点调研：

- 和弦标记位置。
- 段落结构。
- 变调支持。
- 与伴奏生成的连接方式。

风险：

- 不表达旋律和完整节奏。

### iReal Pro 类格式

用途：

- 和弦进行。
- 曲式结构。
- 风格伴奏。

需要重点调研：

- 曲式、重复、结束记号。
- 和弦进行压缩表示。
- 风格伴奏和和弦表之间的关系。

风险：

- 重点是学习产品逻辑和数据建模，不应依赖非公开格式。

## 谱面渲染调研

### alphaTab

alphaTab 适合吉他谱、Guitar Pro 类文件、六线谱 + 五线谱联合显示和播放同步。第一阶段应验证它的 Guitar Pro 导入覆盖率、样式定制能力、自定义数据输入路径，以及与桌面技术栈的集成方式。详见 [alphaTab 渲染调研](alphatab-rendering-research.md)。

### VexFlow

适合方向：

- 五线谱。
- 自定义谱面渲染。
- 细粒度控制。

需要验证：

- 六线谱支持能力。
- 简谱渲染是否需要自定义扩展。
- 口琴谱是否可以作为自定义 notation layer 实现。
- 编辑交互成本。

### OpenSheetMusicDisplay

适合方向：

- MusicXML 渲染。

需要验证：

- 渲染质量。
- 交互编辑能力。
- 与自定义谱面视图的关系。

### MuseScore / LilyPond

适合方向：

- 参考专业排版逻辑。
- 导入导出兼容性调研。

不建议第一阶段作为主交互渲染引擎：

- 它们更偏排版和文件生成，不一定适合实时交互编辑。

## 调性、移调与和声分析

需要支持的核心能力：

- 大调、小调。
- 自然小调、和声小调、旋律小调。
- 五声音阶。
- 布鲁斯音阶。
- 教会调式。
- 临时转调。
- 借用和弦。
- 二级属和弦。
- 罗马数字分析。
- Nashville Number System。
- 首调简谱。
- 固定调唱名。
- 吉他变调夹。
- 口琴不同调与 position。

关键设计点：

- “移调”和“调性分析”不是同一个功能。
- 简谱通常依赖首调体系，需要知道当前 `1 = ?`。
- 吉他谱移调可能涉及重新计算指法，而不是只移动音高。
- 变调夹会改变实际音高和视谱指型之间的关系。
- 口琴谱转换必须考虑口琴调性、孔位、吹吸、压音能力。

## 乐器指法模型调研

### 吉他

需要调研：

- 标准调弦和开放调弦。
- 变调夹。
- 一音多指法选择。
- 和弦指型库。
- 把位优化。
- 可演奏性评分。
- 推弦、滑音、击弦、勾弦、泛音等技巧表示。

核心难点：

- 从五线谱或 MIDI 转六线谱时，同一个音可能有多个弦品位置。
- 自动生成六线谱需要考虑把位连续性和演奏习惯。

### 口琴

需要调研：

- 十孔布鲁斯口琴孔位音表。
- 不同调口琴映射。
- 1st、2nd、3rd position。
- 吹吸关系。
- 压音和超吹的可用音。
- 不同口琴类型的差异。

核心难点：

- 同一个旋律是否适合某个调的口琴，不只看音高，还要看压音难度和 position。
- 口琴谱不能只是音高转换，必须表达吹吸动作。

### 小提琴

需要调研：

- 空弦和把位。
- 指法生成。
- 换弦成本。
- 连弓和分弓。
- 音域和可演奏性。

核心难点：

- 同一个音可以在不同弦和不同把位演奏。
- 自动指法需要考虑音色、换把、换弦和技术难度。

### 钢琴

需要调研：

- 左右手拆分。
- 指法标注。
- 和弦转位。
- 分解和弦伴奏。
- 低音与和声织体。

核心难点：

- 自动伴奏和自动编配要避免只生成机械堆叠和弦。

## 伴奏生成调研

建议先采用基于规则和 pattern 的伴奏系统。

```text
和弦进行 -> 风格 Pattern -> 乐器声部 -> MIDI 事件 -> 音源播放
```

基本原理是：先把和弦进行、拍号、速度、曲式段落等抽象结构变成一组可重复的伴奏 pattern，再按风格规则把 pattern 展开成鼓、贝斯、钢琴、吉他等声部的 MIDI 事件，最后交给音源播放。这样可以保持可控、可编辑、可循环、可变速，并能和谱面小节位置稳定同步。详见 [伴奏生成调研](accompaniment-generation-research.md)。

第一阶段不建议直接做 AI 音频生成。

原因：

- 可控性差。
- 与谱面同步困难。
- 难以编辑和复现。
- 练习软件更需要稳定、可循环、可变速的伴奏。

### 需要支持的伴奏元素

- 鼓。
- 贝斯。
- 钢琴。
- 吉他扫弦。
- 吉他分解。
- Pad。
- 简单旋律填充。

### 风格库方向

- Pop。
- Rock。
- Folk。
- Blues。
- Jazz Swing。
- Bossa Nova。
- Funk。
- Ballad。

### 数据结构重点

- Pattern 的拍号适配。
- Pattern 的小节长度。
- 和弦音选择。
- Bass root / fifth / walking 逻辑。
- Drum groove。
- Strumming pattern。
- Velocity humanization。
- Swing feel。
- Fill-in。
- Intro、ending、transition。

### 可参考产品

- iReal Pro。
- Band-in-a-Box。
- Guitar Pro RSE 播放逻辑。
- DAW 中的 MIDI pattern / arranger track。

## 声卡与实时练习调研

### 目标功能

第一阶段可以先做：

- 选择输入设备。
- 显示输入电平。
- 调音器。
- 实时音高检测。
- 对照谱面判断弹对或弹错。
- 记录节奏偏差。
- 慢速播放和循环练习。
- 延迟校准。

后续再做：

- 多音识别。
- 和弦识别。
- 音色效果器。
- 录音回放。
- 自动生成练习报告。

### 音频技术调研

需要调研：

- CoreAudio。
- ASIO。
- WASAPI。
- JACK。
- Web Audio API。
- AudioWorklet。
- MIDI input。

如果先做 macOS 桌面端，可以优先考虑：

- Web Audio API 做前端交互和基础分析。
- 后续用 Rust 或 C++ 原生模块处理低延迟音频。
- Tauri 或 Electron 承载桌面应用。

### 音高和节奏识别

需要调研：

- YIN。
- pYIN。
- aubio。
- CREPE。
- onset detection。
- beat tracking。
- latency compensation。

核心风险：

- 吉他输入存在噪声、泛音、滑音、推弦和延音，单纯音高检测可能不稳定。
- 多音和弦识别难度明显高于单音识别。
- 声卡、系统、浏览器或桌面框架都可能引入延迟。

建议第一阶段只承诺：

- 单音旋律跟练。
- 调音器。
- 节奏 onset 对齐。
- 简单错误提示。

## 推荐技术栈

### 前端

- React 或 Svelte。
- VexFlow。
- alphaTab。
- Web Audio API。
- Tone.js 或自定义 MIDI 播放层。

### 桌面端

- Tauri 优先。
- Electron 作为备选。

Tauri 优点：

- 体积小。
- 可用 Rust 做后续音频、文件、系统能力扩展。

Electron 优点：

- Web 生态集成成熟。
- 音频和 UI 调试资料多。

### 核心逻辑

建议独立为 `music-core`：

- 音乐模型。
- 调性分析。
- 移调。
- 和弦解析。
- 谱面转换。
- 指法生成。
- 伴奏 pattern 生成。

不要把这些逻辑写死在 UI 组件里。

### 存储格式

建议第一版使用自定义 JSON 项目文件：

```text
project.json
  metadata
  tracks
  sections
  harmony
  notes
  instrument_profiles
  accompaniment
  practice_settings
```

同时逐步支持导入导出：

- MusicXML。
- MIDI。
- ChordPro。
- Guitar Pro 子集。

## MVP 建议

第一版建议聚焦：

```text
吉他练习 + 和弦伴奏 + 多谱显示
```

不要同时完整实现吉他、口琴、小提琴、钢琴。

### MVP 功能范围

1. 手动输入一段旋律和和弦。
2. 内部转换成统一 Music Model。
3. 显示五线谱。
4. 显示吉他六线谱。
5. 显示简谱。
6. 支持移调。
7. 支持变调夹。
8. 支持 MIDI 播放。
9. 自动生成简单鼓、贝斯、和弦伴奏。
10. 声卡输入吉他，提供调音器和单音跟练。

### 暂不进入 MVP 的能力

- 完整 Guitar Pro 文件兼容。
- 完整口琴谱自动生成。
- 小提琴自动指法。
- 钢琴复杂编配。
- AI 生成音频。
- 多音和弦实时识别。
- 专业级混音和效果器。

## 阶段规划

### 阶段 0：技术验证

目标：

- 验证谱面渲染。
- 验证音频输入。
- 验证基础音乐模型。

任务：

- 用 VexFlow 渲染简单五线谱。
- 用 alphaTab 渲染简单六线谱。
- 用 Web Audio 获取麦克风或声卡输入。
- 实现一个简单调音器。
- 实现 `C major` 到 `D major` 的移调。

### 阶段 1：最小产品闭环

目标：

- 用户可以输入、显示、播放、跟练一小段音乐。

任务：

- 建立项目文件格式。
- 建立 `music-core`。
- 实现旋律和和弦输入。
- 实现五线谱、六线谱、简谱显示。
- 实现 MIDI 播放。
- 实现简单伴奏 pattern。
- 实现声卡单音跟练。

### 阶段 2：吉他体验增强

目标：

- 接近基础 Guitar Pro 体验。

任务：

- 改进六线谱编辑。
- 增加吉他技巧记号。
- 增加和弦图。
- 增加把位和指法优化。
- 增加循环练习、变速、节拍器。
- 支持 MusicXML / MIDI 导入导出。

### 阶段 3：口琴与多乐器扩展

目标：

- 在统一模型上扩展不同乐器视图。

任务：

- 增加十孔口琴音表。
- 增加口琴谱渲染。
- 增加不同调口琴和 position 分析。
- 增加小提琴和钢琴的基础视图。

### 阶段 4：高级伴奏和练习分析

目标：

- 更接近 iReal Pro 和 ASD 的练习体验。

任务：

- 增加风格库。
- 增加 intro、fill、ending。
- 增加音频录制和回放。
- 增加节奏偏差报告。
- 增加音准报告。
- 增加练习历史。

## 需要重点调研的问题清单

1. alphaTab 是否足够承担 Guitar Pro 类显示和播放。
2. VexFlow 是否适合扩展简谱和口琴谱。
3. MusicXML 哪些字段必须进入内部模型。
4. MIDI 播放和谱面模型之间如何保持同步。
5. 从五线谱转六线谱时如何选择弦品。
6. 变调夹如何同时影响显示音高和实际音高。
7. 简谱首调体系如何建模。
8. 口琴不同调和 position 如何映射到孔位。
9. Web Audio 的输入延迟能否满足练习需求。
10. pitch detection 在真实吉他输入上的稳定性。
11. 伴奏 pattern 应该如何表达风格、节奏和声部。
12. 第一版用户到底是吉他练习者、口琴练习者，还是泛乐理学习者。

## 当前建议结论

建议第一版以吉他为主线：

- 吉他谱和五线谱联合显示。
- 简谱作为辅助视图。
- 和弦编辑。
- 基础风格伴奏。
- 声卡输入调音和单音跟练。

口琴、小提琴、钢琴暂时作为模型设计时预留的扩展方向，不要作为 MVP 的交付目标。

最关键的工程决策是先做一个稳定的 `music-core`，它应该独立于 UI、桌面框架和具体渲染库。只要这个核心模型设计稳，后续增加谱面、乐器和练习模式都会更可控。
